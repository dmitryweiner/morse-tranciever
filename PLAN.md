## 1. Адаптивное отношение тире/точки в декодере

**Проблема.** `classifyAndAdapt` жёстко предполагает тире = 3×точки: порог
классификации `2×unit`, неоднозначное окно сравнивает `dur` против `dur/3`,
а порог кластеризации — геом. среднее min/max окна (хрупко к выбросам).
Реальная пищалка `samples/TEST.wav` держит тире ≈1.9×точки → «TENT».

**Что делаем** (`src/morse/decoder.ts`):
- Разделять окно меток на кластеры и брать **кластерные средние** (а не
  min/max); порог = геом. среднее средних кластеров.
- Вести оценку `ratio = avgDash / avgDot` (EMA, кламп ~[1.5, 4], дефолт 3);
  обновлять только из окон, где кластера два и в каждом ≥2 метки.
- Классификация в неоднозначном окне: кандидаты `dur` и `dur/ratio`;
  порог точка/тире — `unit × √ratio` вместо жёстких `2×unit`.
- `setUnitMs` (re-seed слайдером) сбрасывает и ratio к 3.

**Тесты:**
- Существующие decoder-тесты остаются зелёными (стандартный 3:1 — поведение
  не меняется).
- Новый: синтетическая «пищалка 1.9:1» декодируется верно после адаптации.
- `tests/samples.test.ts`: закреплённые результаты не хуже; цель —
  `TEST.wav` → «TEST» (если не выйдет — зафиксировать фактический прогресс
  и причину в CLAUDE.md).

**Риски.** Дрейф ratio на шумных окнах — поэтому обновление только из
чистых двухкластерных окон; junk/осколки в ratio не попадают (существующие
guard'ы остаются как есть).

## 2. Передача по тексту (TX)

**Что делаем.** Ввод строки → приложение отстукивает её с точными
PARIS-таймингами и боковым тоном. Заодно получаем генератор посылок для
отладки этапа 4.

- `src/morse/sender.ts` — чистый автомат `TextSender` по образцу
  `PaddleKeyer`: внутри обычный `Keyer` с синтетическими `keyDown/keyUp`
  (события — общий `MorseEvent`, дерево/текст рендерятся тем же
  `applyEvents`). API: `start(text, t)`, `stop()`, `tick(t) → MorseEvent[]`,
  флаг `isToneOn`, прогресс (индекс текущего символа) для UI.
- Паузы: элемент 1u, буква 3u, слово 7u; неизвестные символы
  пропускаются как пробел.
- UI (`src/main.ts` + `index.html`): в TX-панели поле ввода + кнопка
  Send/Stop; во время передачи подсвечивать текущую букву строки; ключи
  на время передачи не блокировать мозгами — просто Stop по любому нажатию
  ключа. Тон — тем же `Sidetone` (ensure() синхронно в жесте).
- UI-тексты — английские.

**Тесты:** длительности элементов/пауз на заданном WPM (скриптовое время,
как в тестах paddle); события идентичны ручному набору той же строки.
Смоук в `scripts/shot.mjs` — передача короткой строки кнопкой.

## 3. Цифры и знаки (без изменения дерева)

**Решение пользователя:** дерево остаётся A–Z (4 уровня); длинные коды
подсвечивают путь до 4-го уровня, дальше — только в строке кода.

- `src/morse/code.ts`: цифры 0–9 (длина 5) и базовые знаки ITU
  (`. , ? / = + - @`, длина 5–6). `MAX_CODE_LENGTH` больше не равен глубине
  дерева — разнести на `TREE_DEPTH = 4` (для `src/ui/tree.ts`) и фактическую
  максимальную длину кода.
- `src/ui/tree.ts`: `setPath`/`flash` не падают и корректно обрезают путь
  на глубине дерева.
- Декодер/кеер работают без изменений (им всё равно, что в таблице);
  `TextSender` из этапа 2 сразу умеет цифры.
- Клавиатурный ввод и глифы `#codeNow` — без изменений.

**Тесты:** таблица против ITU-R M.1677 (спот-проверки), round-trip
encode↔decode, `setPath` с кодом длины 6.

## 4. RX-фронтенд: AudioWorklet + Гёрцель

**Проблема.** AnalyserNode: окно 21 мс + хоп 15 мс размазывают фронты и
квантуют длительности меток крупно; на быстрой морзянке съедаются паузы.
Офлайн-стенд уже считает Гёрцеля — браузер должен считать так же.

**Что делаем.**
- AudioWorklet-процессор (`src/audio/rx-worklet.ts`): копит PCM, каждые
  **5 мс** считает гребёнку Гёрцеля по полосе 300–3400 Гц с шагом ~47 Гц
  (окно ~21 мс, Блэкман — как `wavlab.goertzelPowerDb`), постит
  `SpectralFrame` + точный таймстемп (`currentFrame/sampleRate`) в main.
  Нагрузка ~66 частот × 200 кадров/с — ок, но замерить на мобиле.
- `src/audio/mic.ts`: push-модель вместо `poll()` — колбэк на кадр;
  getUserMedia-констрейнты и синхронное создание контекста не трогаем.
- `src/morse/envelope.ts`: константы в кадрах (`RELOCK_FRAMES`,
  `LOCK_TTL_FRAMES`, `LOCK_MIN_FRAMES`) перевести в **миллисекунды**,
  `SignalGate` принимает hop параметром — иначе смена хопа 15→5 мс
  втихую утроит все выдержки.
- `src/main.ts`: RX-ветка обрабатывает кадры по приходу (со временем кадра,
  не `performance.now`), общий 15-мс цикл остаётся для TX и UI.
- Офлайн-стенд: `runRxChain`/`spectralFrames` на hop 5 мс по умолчанию —
  стенд и браузер снова считают одинаково; пере-закрепить
  `tests/samples.test.ts` на новом хопе (ожидаемо не хуже; слипшиеся точки
  могут начать разделяться — обновить «известные пределы»).
- Пройтись по анти-дребезгу декодера: `GLITCH_MERGE_MS`/`MIN_MARK_UNITS`
  абсолютные или в юнитах — не зависят от хопа, но короткие разрывы,
  которые раньше не были видны на 15 мс, теперь появятся — проверить на
  всех `samples/`.

**Проверка:** unit-тесты gate с параметрическим хопом → `npm run wav` на
всех samples → `npm run rx` (проводка в браузере) → `npm run shot`.

**Риски.** CPU на слабых телефонах (замерить, при нужде проредить гребёнку
вне захвата несущей); поведенческие сдвиги гейта на новом хопе — ловим
регрессией samples до правки констант.
